## 项目概述

SUANXIAN , 贵司名为苏安，生鲜配送，方便叫 苏安鲜 ，这是我第一次完整独立交付的外包项目。需求并不轻：两周时间，要同时做出微信小程序、管理后台和后端 API。技术栈最后落在 Django、Celery、UniApp 和 Vue 3 上，整体开发过程是典型的 SDD 驱动，一边和需求方确认场景，一边把功能拆成能尽快落地的模块。

现在回头看，这个项目真正难的并不是业务本身，而是我第一次在真实交付压力下，把多端开发、构建环境、部署节奏和代码管理全部扛在自己身上。以前写个人项目，很多问题可以绕过去；但做外包不行，任何一个构建细节、线上异常或者交付失误，都会直接变成工期风险。

也正因为这是第一次，我几乎把一整套该踩的坑都踩了一遍。最后项目是交付出去了，但复盘时我最在意的已经不是“做完了什么”，而是“为什么会在这些地方卡住这么久”。

## UniApp 工具链踩坑

项目一开始，我对 UniApp 的命令行生态并不熟。最早的做法很直接：先用 HBuilderX 编译，能跑起来就继续写。这个阶段的问题还不明显，因为开发目标只有一个，就是尽快把页面和流程串起来。

但写到后面我很快意识到，HBuilderX 这条路不适合继续往下走。它对单人快速试错确实友好，可一旦进入需要稳定构建、重复打包和后续维护的阶段，就很难自动化。对交付项目来说，这意味着风险被藏在流程里，只是暂时没有爆出来。

于是我决定迁到 uni CLI。理论上这是更标准的方案，结果真正开始迁移之后，问题反而来了：编译会卡住，而且不是明确报错，而是停在某一步没有任何有效反馈。最折磨人的地方在于，这种故障不像语法错误那样能立刻定位，它更像整个工具链在某个版本组合下悄悄失效。

我后来才查明，问题根源是 uni CLI 当时线上可用版本和本地 Node 版本存在兼容性冲突。这个认知对我来说是个分水岭。之前我对环境的理解还停留在“装上就能用”，从这个项目开始，我被迫真正学会用 nvm 去管理 Node 版本，把运行环境当成项目配置的一部分来维护，而不是电脑里的默认前提。

最后能跑通，不是因为我找到了某个神奇补丁，而是老老实实把变量一个个收紧：切换 Node 版本、核对依赖范围、锁定稳定的 UniApp 版本组合，再重新验证编译链。这个过程非常笨，但有效。它也让我第一次明确感受到，工具选型不是“后面再整理”的事情，项目初期不处理，后面一定会以更贵的方式补回来。

## 生产环境白屏事件

整个项目里最消耗精力的一次故障，是一个只出现在生产环境的白屏问题。更准确地说，它只会在微信体验版的 iPhone 上稳定出现。页面打开以后，tabbar 和顶部栏还在，但主体区域完全空白。开发环境没问题，真机调试也没问题，只有打到那条线上构建链以后才出事。

这种故障最麻烦的地方，不是它“难复现”，而是它复现条件过于苛刻，导致你很难把常规调试手段接进去。我先想到的是上 vconsole，结果它在构建前就失败了，只能拿到一句非常含糊的“页面未注册”。这条信息几乎没有价值，因为它既像路由问题，也像打包问题，还可能只是某个更深层错误的表面现象。

后面几天我基本都在做排除法。埋探针、二分代码、裁剪页面逻辑、替换组件，能试的都试了，但始终拿不到真正的调试信息。因为页面主体根本没有正常挂载，探针放不到有效位置；而二分法在这种“只有生产链路 + 指定容器 + 指定机型”才触发的异常面前，也很快失去效率。那几天最大的挫败感不是 bug 还在，而是你知道自己在努力，却没有任何反馈告诉你方向是对还是错。

真正让我走出来的一步，是把排查对象从“业务代码”切换成“构建链本身”。我后来直接拿微信官方示例小程序，替换掉自己原来的整条构建编译链，反复对照两边产物的表现。也就是说，不再假设问题出在业务实现里，而是把整个链路当成嫌疑对象，从入口开始一层层换掉。

这个思路比继续盯着页面代码有效得多。因为一旦示例项目在某条链路里也出现同样的问题，就能证明故障和业务无关，而是工具链本身的输出有差异。最后我就是靠这种最朴素的对照实验，才把范围压到具体的构建工具链问题上。它没有让我瞬间修好 bug，但至少让我终于知道，自己该怀疑的不是页面逻辑，而是打包结果。

## 经验教训

这个项目还有一个现在看起来很低级、但当时确实交了学费的问题：我第一次在真实开发里用 Git 管理完整项目，习惯并不成熟，结果把密钥带进了 commit 历史。后面即使删掉工作区文件也没意义，因为历史已经污染，只能把仓库转成 private，再按泄漏事件去处理后续风险。

如果说这次复盘最后只留下两条结论，那就是：第一，工具链和环境管理必须在项目初期就认真对待，它们不是“开发完成以后再收口”的附属工作；第二，工程习惯会直接影响交付边界，尤其是版本管理和敏感信息处理，一旦出错，补救成本远高于当下多花的那一点时间。

SUANXIAN 作为第一个外包项目，确实让我吃了很多工具层面的亏。但也正因为这些亏都吃得很具体，我后来再看新项目时，判断标准已经变了。我会更早确认环境、锁版本、设计可替换的构建路径，也会更警惕那些“现在能跑所以先别动”的侥幸心态。对第一个真实交付项目来说，这些教训可能比代码本身更值钱。
