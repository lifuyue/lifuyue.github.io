## 项目概述

TMO 是我接的第二个外包项目，一个面向工业采购场景的一体化平台：商品目录浏览、意向订单、业务员归属绑定、需求收集、售后跟进，这些东西要收束到同一条业务链路里，同时还要覆盖微信小程序、支付宝小程序和 H5 三个端。需求不算轻，但和 SUANXIAN 比起来，这次开发顺利了很多。原因不是项目更简单，而是我换了一套玩法。

技术栈的选择上，这次刻意做了一次主动的矫枉过正。UniApp 在上一个项目让我吃了不少工具链的亏，这次毫不犹豫地换掉了：前端用 Taro + React + TypeScript + Tailwind，原因很直接，Taro 官方文档和示例大抵都指向 React，那就顺势而为。后端大胆换了 Go，开始探索云原生的工具链生态。Go 对我来说是陌生的语言，出发之前也听说各路大模型在 Go 方向上的训练语料远不如 Python 那般丰茂，多少有些顾虑。但这次真正让整个开发节奏变得不一样的，不是语言，而是工作流：大量的实际开发工作，我第一次真正交给了 Agent 来主导完成。

这篇复盘想聊的不是哪个工具链又给我挖了坑，而是在 Agent 工作流全面铺开之后，一些更隐蔽的问题开始浮出水面——自动化工作流在哪里顺畅、在哪里断裂，以及几百个小时下来，我对"人和 Agent 该怎么分工"这件事形成的一些具体认识。

## 后端：测试先行，Agent 闭环

后端的开发流程是这次整个项目里我最满意的一段。它建立在一个反直觉的做法上：先写测试，再让 Agent 写实现。

具体来说，我用的是 SDD（规格驱动）结合 TDD（测试驱动）的方式。在 Agent 写任何代码之前，我先用 Go 标准库的 `testing` 和 `httptest` 把接口行为写成验收用例——响应结构是什么，异常分支怎么处理，边界值是什么。这些用例是可以直接运行的，不是文档里的描述，也不是提示词里的自然语言意图。然后 Agent 的任务就变成一件简单的事：让测试通过。这个转变比看起来要关键得多，因为它消灭了一个长期折磨人机协作的问题——Agent 无法准确理解提示词里那些模糊的业务语义，但它能够准确执行一个明确的合约。接口行为不再依赖"我说清楚了没有"，而是由可执行用例直接锁定。

整个工具链的其他部分也配合这个节奏做了对应的调整。Air 负责热更新，让 Agent 在本地循环里改代码和重跑测试之间不需要手动重启；sqlc 和 oapi-codegen 负责从 SQL schema 和 OpenAPI 规格自动生成数据访问层和接口层的样板代码，这样 Agent 就不用去手工同步这些机械性的重复结构，也减少了在这个环节出错或者漂移的机会。另一个细节是测试 pipeline 的输出配置：只输出失败信息，不打印所有运行日志。这个调整压缩了 Agent 每一轮生成时消耗的 token，同时也让 context window 里有效信息的密度更高，Agent 的后续输出明显更准。

Go 的语料稀疏问题，实测下来其实没有最初担心的那么严重。事后想想，Go 的显式风格——每个类型都要写清楚，错误处理要明确，没有太多语言层面的魔法——反而和 Agent 配合得很好。隐式的、约定俗成的代码结构往往是 Agent 最容易出错的地方，因为它依赖的是训练数据里的模式记忆。Go 的写法本身就倾向于把逻辑平摊在显眼的地方，这让 Agent 在不同轮次之间的输出一致性出乎意料地稳定。真正决定 Agent 输出质量的瓶颈，始终是测试写得够不够清晰，而不是它有没有见过足够多的 Go 代码。

## 前端 Web：从设计稿到代码的加速

Web 管理后台端的开发体验是这个项目里另一段让我比较惊喜的旅程，主角是 Google Stitch 加 Gemini 的这条工作流。

Stitch 是一个 AI 设计稿转代码的工具，过去我对这类工具的评价一直是"原型可以参考，但离生产代码还差一大截"。这次再拿起来用，发现它的输出质量已经迈过了一个门槛——生成的 HTML 结构和样式逻辑已经基本可用，不再是需要大量手工重写的玩具级输出。它的局限是输出格式固定是 HTML，和我们的 React + TypeScript + Tailwind 架构不直接兼容。但 Gemini 可以接手这一步：把 Stitch 输出的 HTML 扔进去，让它转写到我们的技术架构下，转换质量基本可以做到 1:1 还原，不需要大幅人工干预。设计稿进去，React 组件出来，整条链路跑通以后，Web 端的界面开发速度确实快了很多。

这段体验最大的价值不只是省时间，而是让一个人能撑起原本需要设计师和前端工程师配合才能完成的工作量。人还是在，但主要精力转移到了判断"这个组件结构合不合理"和"交互逻辑有没有问题"上，真正机械的转译工作交出去了。这种感觉很好，让人觉得已经找到了一套可以大规模复用的工作流。然后，我把它用在了 Taro 上。

## Taro 跨端：自动化工作流的断裂带

把 Stitch + Gemini 这套工作流直接套用到 Taro 跨端开发上，结果是相当难看的。

问题的根源在于 Taro 的运行环境和普通 Web 根本不一样。H5 和微信小程序的渲染模型存在显著差异，微信和支付宝之间也有各自的平台规则，这些差异在 Web 端完全不存在。设计稿经过 Stitch 输出 HTML、Gemini 转成 React 组件、再缝合进 Taro 仓库，理论上链路是通的，但一到微信或支付宝真机里运行，问题就密集地冒出来：大部分组件的高度会被莫名其妙地压小，布局塌陷，部分交互失效。同样的组件在 H5 里看起来完全正常，切到 App 端就垮掉，再切到支付宝又是另一套问题。那段时间我基本在手动逐个修组件，改了一个又发现下一个，而且修法往往不能通用，得根据平台单独处理。

手动修了一段时间之后，开始出现了一些转机。折磨人的地方往往也是规律最密集的地方。我发现这些平台差异并不是随机的——高度塌陷大多是某几类 flexbox 写法在小程序渲染引擎里行为不一致导致的；某些 CSS 属性在 H5 正常但在 App 端需要显式声明；微信和支付宝对部分事件的冒泡机制处理也有差异。这些规律是可以归纳的，可以成为规则。于是我把它们逐一整理，写进了 `AGENTS.md` 和 `CLAUDE.md`：什么写法要避免，平台差异层应该怎么处理，哪些组件结构需要特殊适配。

这一步做完以后，再重新跑一遍 Stitch + Gemini 的生成流程，把新的源文件缝合进仓库，成功复现率就大幅提升了。Agent 在生成组件的时候已经能够预判平台差异，不再需要每次都手动修一遍同样的问题。从手工调试回到接近全自动的节奏，用的工具没有变，变的是 Agent 掌握的上下文。三端最终实现了 60% 以上的代码复用，微信、支付宝和 H5 从同一套代码库编译产出，这个结果对一个独立开发者来说是能看的。

但这段经历留下了一个我还没有完全回答的问题：这套"踩坑、识别规律、编码进 context、再自动化"的工作流，本身能不能被标准化？它现在还太依赖我对特定平台差异的手工归纳，如果下一个跨端项目来了，这套规则能不能直接复用，或者能不能做成一个更通用的 skill，让这条缝合工作流稳定地、可重复地完成？这个问题我还没想清楚，但觉得值得认真对待。

## 人与 Agent 的协作边界

数百小时和 Codex 协作下来，我对它有了一些比较具体的认识，其中有一条是花了相当长时间才看清楚的：它太能输出胶水代码了，而且对稳定性和兼容性的执念几乎是刻在骨子里的。

具体表现是这样的：在快速迭代阶段，你想要的是大刀阔斧地改，旧数据结构不对就重新设计，旧 API 有问题就重构，没有必要为一个还在频繁变动的系统维护向后兼容。但如果不在 AGENTS.md 里明确告诉它这件事，Codex 默认会帮你在新逻辑和旧逻辑之间搭一堆兼容桥，在新旧数据结构之间写转换胶水，把一个本来该被直接替掉的模块用一层适配包起来继续保留。代码量增加了，但真正有用的部分没有增加，技术债反而在悄悄积累。这是它的一种"善意"，但在迭代速度优先的阶段，这种善意适得其反。

后来我在 AGENTS.md 里加了一句话："项目还处于快速迭代阶段，鼓励大规模重构，不用考虑兼容旧数据。"效果是实实在在的。后续的生成明显干净了很多：该删的逻辑直接删，该重写的模块重写，不再出现那些为了维护兼容性而存在的冗余层。这和我后来读到的一份公开的 Claude Code 最佳实践合集里的最后一条不谋而合——和 Agent 的协作，积累的不只是技巧，而是对它的特性和习惯的具体认识，这种认识本身就是一种经验资产。知道在哪里推它一把、在哪里给它划边界，比会写漂亮的提示词更值钱。

从这个项目的整体经历来看，人的角色在工作流里已经明显移位了。后端大量的实现工作交给了 Agent 的测试驱动闭环；前端的界面转译交给了 Stitch + Gemini；自动化测试的执行和归因交给了通过 MCP 打通 Chrome DevTools 的 Codex。一个人用原来需要小团队才能覆盖的工作量完成了这个项目。但我花时间最多的地方，是在写测试、识别跨端规律、向 AGENTS.md 里补规则——这些都是在为 Agent 创造它能够有效运作的条件，而不是替它写代码。

说到底，SUANXIAN 教会了我工具链的纪律，TMO 教会了我协作的纪律。前者是学会控制环境，后者是学会控制边界。成百上千小时的 Codex 时间和真金白银的 token 消耗最后换来的，不只是一个能交付的产品，还有一套关于怎么和 Agent 真正配合起来的更具体的认识——哪些判断必须由人来做，哪些事情可以完全放手，以及如何让这两件事之间的摩擦尽可能地小。
